# Service Boundary Validation — LMS Backend Service Map

issue_type | affected_service | description | severity | suggested_fix
--- | --- | --- | --- | ---
Domain overlap | user_profile_service | `RoleAssignment` appears authorization-scoped, while role/permission ownership is typically part of identity/access control rather than learner profile metadata. This creates overlap with identity/authorization concerns. | high | Move role/permission assignment ownership to a dedicated authorization service (or auth_identity_service), and keep user_profile_service focused on learner profile attributes (employee metadata, preferences, status).
Domain overlap | user_profile_service + organization_catalog_service | `Department` appears in user_profile_service while organization_catalog_service owns org structure (`BusinessUnit`, `Team`, `Location`, `LearnerGroup`). Department is an org entity and should have a single source of truth. | high | Make organization_catalog_service authoritative for Department (or rename to OrgUnit). user_profile_service should reference department_id only.
Unrelated entity ownership | learning_path_service | `CurriculumAssignment` in learning_path_service mixes path definition with learner/target assignment lifecycle, which is operational enrollment/assignment orchestration rather than structure definition. | medium | Keep learning_path_service responsible for path topology and prerequisite rules only; move assignment entities/workflows to enrollment_service (or an assignment orchestration service).
Domain overlap | content_delivery_service + course_catalog_service | content_delivery_service owns `ContentVersion`, while course/catalog also manages publish state and discoverability. Versioning and publish lifecycle can become split-brain across delivery vs catalog/content management concerns. | medium | Clarify ownership: either delivery owns only secure asset serving/access grants, while catalog/content owns version lifecycle; or introduce a dedicated content repository/versioning service and make both services consume it.
Domain overlap | progress_tracking_service + reporting_analytics_service | progress_tracking_service captures `LearningEvent`, which can overlap with analytics event ingestion ownership and duplicate telemetry pipelines with reporting_analytics_service aggregates. | medium | Define explicit split: progress_tracking_service computes authoritative learner progress state; analytics ingests immutable event streams for BI/KPI use. Ensure event schema contract and single event producer path.
Cross-cutting concern embedded in domain map | notification_service | Notification capability is modeled as a business service but is typically platform/integration cross-cutting infrastructure used by multiple domains. Risk: duplicated notification logic if other domains also notify directly. | low | Reframe notification_service as Notification Core/Connector platform service with domain-owned notification intents/events.
No issue | auth_identity_service, course_catalog_service, enrollment_service, assessment_service, certification_service | These services have generally coherent single responsibilities in the current map, assuming clear API contracts and ownership boundaries are documented. | info | Keep as-is; add explicit ownership matrix in architecture docs to prevent future drift.
