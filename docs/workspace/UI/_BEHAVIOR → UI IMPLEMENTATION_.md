========================================================
1. ACTION-FIRST UI (GLOBAL RULE)
========================================================

SOURCE:
§2.1 Action-First Interaction
§1.1 System Acts as Operator

RULE:
Every screen must surface PRIMARY ACTION first.
The first visible element is always what to do — not what exists.

IMPLEMENTATION:
- Use Card + Button (shadcn)
- First visible element = action, never a metric alone

PATTERN:

  Card:
    Title: "5 unpaid fees"
    Action: [Send Reminders]

  NOT:
    Card:
      Metric: "PKR 45,000 outstanding"
      (no action)

PROHIBITED:
- Dashboards with only metrics and no actions
- Passive data views that require user interpretation
- Any screen whose first element is a chart, table, or count

========================================================
2. DAILY ACTION LIST (CORE COMPONENT)
========================================================

SOURCE:
§5.1 Daily Action List
§4 Decision Reduction

COMPONENT:

<ActionList />

STRUCTURE:
- Priority groups (system-ranked, never user-configurable):
  - Critical   — requires immediate attention
  - Important  — should be completed today
  - Optional   — context available on demand

UI:
  Accordion (shadcn)
  List + Badge + Action Button

BEHAVIOR:
- Auto-generated by system on every session start
- Auto-sorted by system priority — user cannot reorder
- First item = most important action at that moment
- Always visible on entry — not behind a nav item
- Must include: unpaid fees, absent students, inactive users,
  pending approvals, urgent alerts, revenue at risk

PROHIBITED:
- User-defined sort order on the action list
- Hiding the action list behind a "Reports" or "Insights" nav item
- Showing the list without actionable buttons on each item
- Empty action list UI — if no actions, show "All clear" state, not blank

========================================================
3. CONTEXTUAL ACTIONS (SMART CONTROLS)
========================================================

SOURCE:
§4.1 System Suggests Actions
§7.2 Contextual Upsell
§3.2 Event-Driven Behavior

RULE:
Actions appear ONLY when relevant context exists.
System detects state changes and surfaces actions automatically — user does not hunt for them.

IMPLEMENTATION:

  IF unpaid_fees > 0:
    show Button("Send Reminder")
  ELSE:
    hide button entirely (not disabled — hidden)

  IF student_missed_sessions >= 3:
    surface action item: "Batch B — 3 absences — [Send Alert]"

  IF batch_performance_declining:
    surface action item: "Batch A performance down 12% — [Schedule Intervention]"

SHADCN:
- Conditional Button
- Tooltip for explanation when action context is non-obvious

EVENT TRIGGERS (system watches these, user does not configure):
- Fee overdue → action item
- Attendance threshold missed → action item
- Engagement drop detected → action item
- Pending approval aging → escalated action item
- Sync failure → recovery action item

PROHIBITED:
- Disabled buttons waiting for user to "enable" a feature
- Showing actions that have no applicable context at that moment
- Requiring user to navigate to a state before seeing the relevant action

========================================================

SOURCE:
§4.2 Prioritization Engine
§15.1 Reduce Cognitive Load

RULE:
System ranks everything. User never decides order.
Most important action is always first, always visible.
Lists filtered to relevance by default — raw unfiltered view requires explicit opt-in.

PRIORITY HIERARCHY (system-enforced, not user-configurable):
  Critical  → red    (requires immediate action)
  Important → yellow (should be done today)

IMPLEMENTATION:
- All lists arrive pre-sorted by system priority
- Critical items surface at top regardless of list type
- No "sort by" dropdown on the Daily Action List
- Optional items collapsed by default (Collapsible/Accordion)

PROHIBITED:
- User-facing sort controls on action lists
- Presenting all priority tiers with equal visual weight
- Showing optional items at the same prominence as critical items
- Unfiltered raw data as the default view — user must explicitly opt in to "see all"

========================================================
9. ERROR HANDLING UI
========================================================

SOURCE:
§10.4 No Error State Without a Resolution Path

RULE:
Every error must include:
1. Plain-language description of what went wrong (no error codes, no stack traces)
2. Specific resolution step the user can take immediately
3. Fallback path if self-resolution is unavailable

UI:
  Alert (shadcn) with action button

EXAMPLES:
  "Payment failed — the gateway timed out.
   [Retry Payment]  or  [Contact Support]"

  "Video upload failed — file size exceeds 500MB.
   [Compress and retry]  or  [Upload a smaller file]"

  "Sync failed — no internet connection.
   We've saved your changes locally. [Retry when connected]"

FORM VALIDATION RULE:
  Every field error must name the specific field AND state the required format.

  CORRECT:   "Student email — enter a valid email address (e.g. name@domain.com)"
  WRONG:     "Invalid input"
  WRONG:     "Error code 422"

REQUIRED before release:
  All user-facing error states must be pre-mapped to plain-language resolution templates.
  No error code or system exception message may surface to a non-technical user.

PROHIBITED:
- "Something went wrong. Please try again." with no specifics
- HTTP status codes visible to users (404, 500, 422, etc.)
- Silent failures where an action appears to succeed but has not
- Errors without at least one actionable resolution step

SOURCE:

COMPONENT:

<SyncStatusIndicator />

STATES:
  ● synced    — green dot, "All saved"
  ↻ syncing   — animated icon, "Syncing..."
  ✕ offline   — amber badge, "Offline — X changes pending"

SHADCN:

BEHAVIOR:

- Silent sync failures — every failed sync must notify the user with a recovery path
- Requiring user to manually download content before going offline

========================================================
11. HIGH-STAKES MODE (EXAMS)
========================================================

SOURCE:
§10.3 High-Stakes Session Behavior

RULE:
During an active exam session, the platform operates in stability-prioritized mode.
No operation may interrupt or degrade an active exam session.

UI:
- Distraction-free layout (no nav, no sidebar, no notifications)
- Locked navigation (back button disabled or intercepted with confirmation)
- Progress indicator (question X of Y) — no other status chrome

ANSWER PERSISTENCE:
- Every answer saved to server on question submission — not only at final submit
- Local draft saved every 30 seconds as fallback
- On reconnect after drop: session resumes from last server checkpoint — no answer loss

SHADCN:
- Dialog (for reconnection notice — not a blocking modal, a soft indicator)
- Progress (question progress bar only)

PROHIBITED:
- Modals or toasts during an active exam (exception: reconnection notice only)
- Background software updates during exam session
- Capability gate interruptions mid-exam (plan lapse enforcement deferred to post-session)
- Auto-sync operations competing with exam bandwidth
- Forced session re-authentication during exam
- Any answer loss on network interruption

14. INSIGHTS UI (NOT REPORTS)
========================================================

SOURCE:
§14.1 Insights Over Reports
§14.2 Comparative Context

RULE:
Every data surface must answer "what should I do next?" — not "here is what happened."
Every chart must include: insight + comparative context + action.

PATTERN:
  Chart — Attendance Trend:
    Value:   "72%"
    Context: "Down 8% from last week"

  Always pair with: vs last period / vs benchmark / vs target.

SHADCN:
- Card (for insight wrapper)
- Button (for action)

PROHIBITED:
- Charts with no insight label

- Reports accessible only via deliberate navigation — insights surface proactively

18. EVENT-DRIVEN STATE TRIGGERS
========================================================

SOURCE:
§3.2 Event-Driven Behavior
§4.1 System Suggests Actions

RULE:
System watches state changes and surfaces actions automatically.
User does not hunt, filter, or configure to see what needs attention.

TRIGGER TABLE (system-managed, not user-configurable):

  Event                          → UI Output
  ─────────────────────────────────────────────
  Fee overdue                    → Action item in Daily List + notification
  Student misses 3 sessions      → Alert item + [Send Alert] action
  Batch performance drops 10%+   → Insight card + [Schedule Intervention]
  Pending approval > 24h         → Escalated Critical item
  Sync failure                   → Persistent badge + recovery path
  Payment received               → Instant access activation + confirmation toast
  Learner inactive 7 days        → Engagement risk item + [Re-engage] action
  Exam session starts            → Platform enters High-Stakes Mode automatically

SHADCN:
- Toast (transient events)
- Badge (persistent state indicators)
- Alert (critical state changes)

PROHIBITED:
- Requiring user to run a report to discover any of the above states
- Event triggers that require manual configuration to activate
- State changes that update silently without any UI signal to the operator

========================================================

SOURCE:
§5.4 Multi-Branch Operations

RULE:
HQ-level users see aggregated cross-branch status by default.
Branch operators see only their branch.
Same interface — different data scope determined by RBAC role.

PROHIBITED:
- Requiring HQ operator to log into each branch separately
- Cross-branch data available only via a manually generated report
- Single-branch permission model applied to multi-branch roles
- Branch selector visible to branch-scoped operators

20. PAYMENT ACTIVATION FLOW
========================================================

SOURCE:
§7.3 Payment Confirmation Triggers Instant Automatic Access

RULE:
Zero manual steps between confirmed payment and activated access.
Payment confirmation → entitlement activation → learner notification.
Entire sequence is automatic.

FLOW:

  Payment callback received
    → System confirms transaction
    → Access activated immediately
    → Learner notified in-channel (WhatsApp / email / in-app)
    → Operator sees activity log entry

  On failure:
    → Learner sees: what failed + specific resolution step + fallback path
    → Operator sees: failed payment item in Daily Action List + [Retry] or [Verify Manually]

UI STATES:
  Processing:  "Confirming your payment..."  (Loader + Progress)
  Success:     "Access activated — [Start Learning]"  (Toast + CTA)
  Failure:     "Payment could not be confirmed. [Retry]  [Contact Support]"  (Alert)

SHADCN:
- Progress (processing state)
- Toast (success)
- Alert (failure with action)

PROHIBITED:
- Manual payment reconciliation (screenshot upload, operator "mark as paid" for standard flows)
- Delayed activation pending "payment review" for standard payment callbacks
- Silent hold on failed payment — must present resolution path immediately
- Access remaining locked after payment is confirmed

26. FAILURE RESILIENCE UI
========================================================

SOURCE:
§10.1 Failure Resilience
§15.2 Build Trust Through Consistency

RULE:
Failures must never result in silent data loss.
System retries automatically and syncs when connectivity is restored.
User is always informed — never left wondering if their action succeeded.

UI STATES:

  Retry in progress:
    Subtle spinner + "Retrying..." — not a blocking modal

  Retry exhausted:
    Alert: "Could not save. [Try again]  [Save a copy]"

  Auto-save active:
    Persistent soft indicator: "All changes saved" / "Saving..."

  Action outcome confirmation:
    Every destructive or important action must confirm success visibly:
    Toast: "Reminder sent to 3 students"  ← positive confirmation
    NOT:   (silent — user left to assume it worked)

SHADCN:
- Toast (success confirmation)
- Alert (failure with recovery action)
- Badge (persistent save state indicator)

PROHIBITED:
- Silent data loss on any user action
- Actions that appear to succeed but have not (no silent failures)
- Blocking modals for retry states — must be non-intrusive
- Missing success confirmation on important actions

27. ONBOARDING / INSTANT START
========================================================

SOURCE:
§12.1 Instant Start
§15.3 Provide Quick Wins Early

RULE:
New user must reach a meaningful outcome within their first session — without manual setup.
Onboarding must not defer value until full configuration is complete.

MEANINGFUL FIRST-SESSION OUTCOMES (any one of):
  - First course published
  - First student enrolled
  - First reminder sent
  - First payment received

  [Then] start using the platform

SHADCN:
- Progress (step indicator)
- Card (per setup step — shows result immediately)
- Toast (first-action confirmation: "First student enrolled!")

PROHIBITED:
- Onboarding that completes without the user experiencing any outcome
- Setup screens that must be fully completed before the platform is usable
- Deferring the first automated action until after onboarding is finished

========================================================
28. SMART DEFAULTS
========================================================

SOURCE:
§13.2 Smart Defaults
§2.2 Minimal Input

RULE:
System pre-fills, auto-configures, and reuses established patterns.
A new operator must complete a meaningful workflow without reading documentation.

IMPLEMENTATION:

  Pre-fill from context:
    - Fee reminder message: load last-used template
    - Attendance session: pre-fill with current batch + today's date
    - New course: load last-used category, duration estimate, content type

  Auto-configure:
    - Timezone from device / browser
    - Currency from operator's registered country
    - Default notification channel from most-used channel

  Reuse patterns:
    - "Use same settings as last batch" affordance on batch creation
    - Duplicate course as starting point for new content

SHADCN:
- Combobox with pre-selected default value
- Form fields with placeholder = last-used value
- "Use previous settings" checkbox on repeat-action flows

PROHIBITED:
- Forms that discard context and require full re-entry each time
- Configuration options with no sensible defaults
- Any capability requiring 10+ setup steps before it is usable

========================================================
29. COMPARATIVE CONTEXT (ANALYTICS)
========================================================

SOURCE:
§14.2 Comparative Context
§14.1 Insights Over Reports

RULE:
Never show an absolute number alone.
Always pair with: vs last period / vs benchmark / vs target.

PATTERN:

  CORRECT:
    "Attendance: 72%  ↓8% vs last week"
    "Fees collected: PKR 45,000  ↑12% vs last month"
    "Completion rate: 68%  — below 80% target"

  WRONG:
    "Attendance: 72%"
    "Fees: PKR 45,000"
    "Completion: 68%"

REQUIRED for every metric display:
  1. Current value
  2. Comparison period (last week / last month / target)
  3. Direction indicator (↑ ↓ → )
  4. Action if below target or declining

SHADCN:
- Card (metric wrapper)
- Badge (trend direction: up / down / stable)
- Button (action when metric is below threshold)

PROHIBITED:
- KPI cards showing only the current value
- Charts with no baseline or benchmark reference
- Trend data without direction language

========================================================
30. COGNITIVE LOAD + FILTERING
========================================================

SOURCE:
§15.1 Reduce Cognitive Load

RULE:
Never present more information than the user needs to take the next action.
Lists and dashboards filtered to relevance by default.
Raw unfiltered views require explicit user opt-in.

IMPLEMENTATION:

  Default state (always):
    - Action list shows only items requiring attention today
    - Student list shows only active students
    - Course list shows only published + in-progress courses

  Unfiltered / raw state (explicit opt-in only):
    - "Show all" toggle — not the default
    - Clearly labeled: "Showing all records — [Return to active view]"

  When multiple actions are possible:
    - System ranks and presents most important first
    - Secondary actions behind "More" or Collapsible — not shown at equal prominence

SHADCN:
- Toggle (active/all filter)
- Collapsible (secondary actions)
- Select / Combobox (filter by batch, date, status — always defaults to "current / active")

PROHIBITED:
- Unfiltered raw data as the default view in any list
- All priority tiers shown with equal visual weight
- Multiple equally prominent actions presented simultaneously

========================================================
31. TRUST + VALUE SIGNALS
========================================================

SOURCE:
§15.2 Build Trust Through Consistency
§15.4 Reinforce Value Continuously

TRUST RULES:
  - Same action must produce the same result regardless of channel, device, or interface
  - System messages must use consistent plain language — no technical codes to non-technical users
  - Every automated action must be visible, reversible, and logged

VALUE REINFORCEMENT (proactive):
  System must surface moments of value without being asked:
    "You collected 12 fees this week — PKR 54,000 recovered"
    "Batch A completed 3 modules this week — on track for certification"
    "Reminder sent automatically to 5 students — 3 have already paid"

  Delivery mechanism:
    - In-app: Toast or subtle banner (not a modal)
    - Timing: end of session or on meaningful threshold (10th enrollment, first payment, etc.)

SHADCN:
- Toast (value moment notification)
- Card (weekly summary card in dashboard)

PROHIBITED:
- Value moments buried in reports (must surface proactively)
- Automated action summaries that require navigation to find
- Inconsistent outcomes for the same action across channels

========================================================
32. COPY RULES — BUSINESS IMPACT LANGUAGE
========================================================

SOURCE:
§7.6 Capability Introduction Uses Business Impact Language

RULE:
ALL capability introductions, onboarding prompts, upgrade paths, gate messages,
help text, and action labels must use business impact language.
Never use feature names, technical keys, or system identifiers in user-facing copy.

REQUIRED (Business Impact) → PROHIBITED (Feature Language):

  "Automatically remind students when fees are overdue"
  → NOT: "Enable: auto_reminder_capability"

  "Protect your video content from screen recording"
  → NOT: "Enable: media_security_watermark_v2"

  "Let students pay instantly via JazzCash — access their course immediately"
  → NOT: "Configure: payment_adapter_jazzcash"

  "See which students are at risk of dropping out before it happens"
  → NOT: "Enable: learner_risk_analytics"

  "Run attendance from WhatsApp — no portal required"
  → NOT: "Enable: whatsapp_interaction_layer"

  "Recover unpaid fees automatically"
  → NOT: "Upgrade to access automation features"

COPY CHECKLIST (apply to every string before release):
  □ Does it say what the outcome is for the user?
  □ Does it avoid technical key names?
  □ Does it avoid "Upgrade Required" without context?
  □ Does it use plain operational language, not accounting/technical terminology?
  □ If it's an error, does it say what failed AND what to do?
  □ If it's an action label, does it say what will happen, not just a generic verb?

ACTION LABEL RULES:

  CORRECT:   [Send Reminder]  [Collect Now]  [Start Course]  [Mark Attended]
  WRONG:     [Submit]  [OK]  [Confirm]  [Execute]  [Process]

  Button copy must complete the sentence: "I want to ___"
  If the label doesn't complete that sentence clearly, rewrite it.

PROHIBITED:
- Capability key names in any user-facing surface
- "Something went wrong" as a complete error message
- "Upgrade Required" without value demonstration
- Generic action labels (OK, Submit, Confirm) where a specific verb is possible
- Accounting or financial system terminology in operational surfaces

========================================================
FINAL LAW
========================================================

The UI must:

  SHOW ACTION   — every screen answers "what do I do next?"
  REDUCE THINKING — system ranks, filters, and suggests; user confirms and acts
  AUTOMATE FLOW — repetitive tasks run by default; user opts out, never in

The UI must NOT:

  DISPLAY DATA ONLY    — data without action is incomplete
  REQUIRE NAVIGATION   — daily operations reachable without knowing module structure
  TRUST THE USER       — to know what matters, what order to work, what to configure
  EXPOSE COMPLEXITY    — advanced config is secondary layer, never in primary flow
