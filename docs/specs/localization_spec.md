# Localization Architecture

component | localization_data | usage
--- | --- | ---
Language Packs | Versioned JSON/YAML bundles per locale (e.g., `en-US`, `fr-FR`, `ar-SA`) containing namespace-scoped translation keys (`common.*`, `course.*`, `assessment.*`), fallback chain metadata (`fr-CA -> fr -> en`), and pluralization rules tied to ICU message syntax. | Loaded at app startup for core locales and lazy-loaded on language switch for secondary locales. Backend APIs expose locale + pack version so clients can cache and invalidate correctly. Missing keys fall back to parent locale, then platform default.
Regional Formats | Locale policy registry storing CLDR/ICU-based date/time formats, number/currency separators, timezone defaults, week-start conventions, grading scale display rules, and measurement preferences. Includes tenant-level overrides for region-specific compliance formatting. | Applied by a shared formatting service/UI utility layer (`formatDate`, `formatCurrency`, `formatNumber`, `formatDuration`) so rendering is consistent across web, mobile, emails, and reports. Server-side export jobs use the same registry to keep PDFs/CSVs aligned with UI output.
Translation Keys | Central key taxonomy with domain prefixes (`nav.home`, `course.module.locked`, `notification.assignment_due`), developer comments/context, interpolation placeholders, and status metadata (`draft`, `reviewed`, `approved`). Stored in TMS-compatible schema with key ownership and change history. | Referenced in code via typed i18n helpers instead of hardcoded strings. CI lint checks enforce key existence, placeholder compatibility, and orphan detection. Product/content teams update translations in TMS; deployment pipeline syncs approved keys into language packs.
