Service map misalignment | tenant_service | Stage 2 defines a standalone `tenant_service_spec`, but the Stage 0 service map has no tenant service and instead places organization and audience ownership in `organization_catalog_service`. This creates an untracked service boundary expansion relative to Stage 0. | high | Either (a) add `tenant_service` to the canonical service map with explicit ownership split from `organization_catalog_service`, or (b) fold tenant lifecycle responsibilities into `organization_catalog_service` and deprecate the standalone spec.
Responsibility overlap | auth_service + user_service | `auth_service` owns authentication/SSO/session/token lifecycle, but `user_service` also owns identity-link operations (`map_external_identity`, `unmap_external_identity`) and account lock/unlock transitions that directly affect authentication eligibility. This duplicates identity/security ownership. | high | Move identity federation mapping and authentication eligibility controls (lock/unlock) into `auth_identity_service`; keep `user_service` focused on profile/lifecycle metadata.
Responsibility overlap | course_service + lesson_service + content_service | Course publishing/versioning, lesson publishing, and content retrieval/storage are split across three specs without a clear handoff contract. `course_service` versions content refs, `lesson_service` controls lesson publish visibility, and `content_service` controls asset access, creating ambiguous publishing authority. | medium | Define a single publish authority model: course/lesson manage pedagogical structure states, while content service is only asset repository/delivery. Add explicit state machine and API ownership matrix for publish transitions.
Stage 0 alignment gap | enrollment_service | Stage 0 identifies `enrollment_service` as a core service, but Stage 2 does not provide a dedicated enrollment service specification in the service-spec set, while assignment-related fields appear in learning path/dashboard artifacts. This leaves enrollment ownership under-specified. | high | Add an `enrollment_service_spec` defining enrollment, waitlist, approval, and status-transition ownership; remove assignment lifecycle behavior from adjacent services.
Domain boundary mismatch | learning_path_service | `learning_path_spec` includes `path_assignment_scope` and assignment targeting (`role|department|location|manual`), while Stage 0 defines learning path service for sequencing/prerequisites and enrollment service for learner assignment lifecycle. Assignment scope inside learning path crosses into enrollment/orchestration boundaries. | medium | Keep learning path service responsible for path topology and completion rules only; route assignment targets and activation into enrollment service APIs/events.
Domain boundary mismatch | analytics/event_ingestion vs progress_tracking_service | Stage 2 defines an `event_ingestion` pipeline that collects and validates events for analytics storage, while `progress_tracking_service` is the progress event authority. Without explicit producer/consumer contracts, event ownership between analytics ingestion and progress state computation can overlap. | medium | Declare progress tracking as system-of-record for learner progress state/events, and constrain event ingestion to analytics pipeline ingestion/validation only. Add event ownership table (producer, canonical schema owner, downstream consumers).
Service naming inconsistency | analytics_reporting_service vs reporting engine | Stage 0 service map uses `reporting_analytics_service`, while Stage 2 introduces separate reporting/manager dashboard artifacts that reference multiple service-owned datasets with inconsistent service naming patterns. This weakens boundary clarity and ownership traceability. | low | Normalize service IDs across Stage 0 and Stage 2 (e.g., `analytics_reporting_service`) and require all specs to reference canonical service names in dependencies.
